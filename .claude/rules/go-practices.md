# Go コーディングプラクティス

## インフラとアプリの分離

- Go（`provided.al2023`）で記述し、デプロイには必ず `lambroll` を使用する
- Terraform 内で Lambda のコードをビルドしない
- インフラ（器）とアプリ（中身）のデプロイライフサイクルを完全に分離する

## 原則

- **純粋関数を優先**: 副作用のない関数から実装し、IO 操作は関数の境界に押し出す
- **不変データ構造**: 値オブジェクトは不変とし、制御された更新のみ許可する
- **副作用の分離**: 外部依存はインターフェースで抽象化し、テスト時はインメモリ実装で差し替える
- **早期リターン**: 条件分岐をフラット化し、ネストを最小限にする

## エラー処理

- `error` を返す関数は、呼び出し元で必ずエラーチェックする
- エラーは `fmt.Errorf("context: %w", err)` でラップし、コンテキストを付与する
- 標準出力（`stdout`）と標準エラー出力（`stderr`）を明確に分離する
  - 正常な処理結果 → `stdout`
  - エラーメッセージ・デバッグ情報 → `stderr`
  - Rundeck 等のジョブスケジューラでログを正しく分類するため
- パニックは使用せず、エラーを値として返す

## 静的解析

- **golangci-lint**: Go コードの静的解析に使用する
- **trivy**: 依存パッケージの脆弱性スキャンに使用する
- **Biome**: Go 以外のフロントエンド関連ファイルの Lint/Format に使用する

## テスト

- テーブル駆動テストを優先する
- テスト名は「状況→操作→結果」の形式で記述する（例: `TestCreateUser_InvalidEmail_ReturnsError`）
- インターフェースを活用してテスト用のモック・スタブを容易に差し替え可能にする
- テストヘルパーには `t.Helper()` を付与する
